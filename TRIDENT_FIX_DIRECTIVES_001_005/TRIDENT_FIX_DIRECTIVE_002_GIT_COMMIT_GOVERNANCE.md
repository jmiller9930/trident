# TRIDENT FIX DIRECTIVE 002
## Git Commit Governance Enforcement

---

## 1. Purpose

Close the Git governance gap by ensuring local repository changes cannot drift silently from Trident-managed task state.

---

## 2. Problem

Users can commit, reset, checkout, or mutate Git state outside Trident. This can break task proof, file lock ownership, and backend state.

---

## 3. Required Fix

Engineering must implement Git governance enforcement for Trident-managed projects.

Required behavior:

- Detect branch state before work begins.
- Detect dirty state continuously during active directives.
- Require diff capture before task review.
- Require commit hash as proof for accepted work.
- Detect external Git changes and mark task state as `GIT_DRIFT_DETECTED`.
- Provide remediation flow before task closure.

---

## 4. Required Controls

Engineering must implement or document:

- backend Git status polling
- IDE Git event monitoring
- optional pre-commit hook generated by Trident
- optional commit-msg hook to require directive ID
- backend validation before closure

---

## 5. Acceptance Criteria

- Trident detects out-of-band Git commits.
- Trident blocks closure if Git state is inconsistent.
- Commits can be tied to directive IDs.
- Git drift creates an auditable event.

---

## 6. Required Tests

- clean repo validation
- dirty repo detection
- external commit detection
- branch switch detection
- commit without directive ID rejection or warning
- closure blocked on Git drift

---

## 7. Proof Objects

Engineering must return:

- Git drift logs
- pre-commit / commit-msg hook proof if implemented
- closure-block proof
- commit hash proof
- audit samples
